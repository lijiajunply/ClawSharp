# 功能规范：SpecKit 脚手架集成与自举开发

**功能分支**: `luckyfish/002-speckit-scaffolder-integration`  
**创建日期**: 2026-03-31  
**状态**: 草案  
**输入**: 用户描述："自动化与脚手架 (SpecKit 增强) 项目中存在 .specify 文件夹，包含了很多任务管理和规划的模板。 - 改进建议： - 内核集成：将这些 Spec 工具直接集成到 IProjectScaffolder 中。目前 /init-proj 只是简单的文件复制。 - 自举开发：利用 Planner Agent 配合 git_ops 和 file_write 工具，实现自动根据 plan.md 生成任务分支、创建占位文件并更新 tasks.md 的闭环。"

## 用户场景与测试 *(必填)*

### 用户场景 1 - 带有 SpecKit 的完整项目初始化 (优先级: P1)

作为一个开始新项目的开发者，我希望项目初始化命令不仅能设置代码结构，还能设置完整的 SpecKit 治理框架，以便我可以立即开始使用 AI 驱动的规划和实施。

**优先级理由**: 这是“SpecKit 优先”开发模型的基础。如果没有集成的脚手架，用户必须手动复制 `.specify` 文件夹，这很容易出错。

**独立测试**: 可以通过运行项目初始化命令（如 `/init-proj`）并验证新项目中是否正确存在并配置了 `.specify` 目录及其所有模板来进行测试。

**验收场景**:

1. **给定** 一个新项目目录，**当** 我运行初始化命令时，**那么** 会创建一个包含 `templates/`、`scripts/` 和 `memory/` 子目录的 `.specify` 目录。
2. **给定** 一个已初始化的项目，**当** 我检查 `spec-template.md` 时，**那么** 它应该包含标准的 ClawSharp 规范结构。

---

### 用户场景 2 - 从计划自动生成功能脚手架 (优先级: P1)

作为一个刚刚完成新功能 `plan.md` 的开发者，我希望 "Planner Agent" 能够自动将该计划转化为物理项目结构（分支、文件夹、文件）和跟踪任务。

**优先级理由**: 这实现了“规划”到“执行”的闭环。它减少了手动的模板工作，并确保实施严格遵循计划。

**独立测试**: 可以通过向 Planner Agent 提供有效的 `plan.md` 并验证是否创建了新的 Git 分支、生成了指定的文件/目录，以及 `tasks.md` 是否更新了计划的里程碑来进行测试。

**验收场景**:

1. **给定** 功能目录中有一个完成的 `plan.md`，**当** 系统检测到文件保存并询问“是否根据此计划生成脚手架？”且我确认后，**那么** 会使用计划中定义的短名称创建一个新的 Git 分支。
2. **给定** 一个包含“组件”或“文件”列表的 `plan.md`，**当** 脚手架运行时，**那么** 所有列出的文件都会被创建为占位符（例如，空类或样板代码）。
3. **给定** 一个包含“任务”或“里程碑”的 `plan.md`，**当** 脚手架运行时，**那么** 会生成或更新 `tasks.md` 文件，并将这些项目标记为未完成。

---

### 用户场景 3 - 集成的 SpecKit CLI 命令 (优先级: P2)

作为一个开发者，我希望直接通过 ClawSharp CLI 与 SpecKit 工具交互（例如 `/speckit plan`，`/speckit implement`），而不是手动运行 shell 脚本。

**优先级理由**: 改善开发者体验，使 SpecKit 工作流易于发现和使用。

**独立测试**: 可以通过在 CLI 中执行 `/speckit` 命令并验证它们是否触发了相应的内部逻辑或脚本来进行测试。

**验收场景**:

1. **给定** ClawSharp CLI，**当** 我输入 `/speckit` 时，**那么** 我会看到可用规范驱动开发命令的列表。

## 需求 *(必填)*

### 功能需求

- **FR-001**: `IProjectScaffolder` 必须支持“规范感知”初始化，自动将 `.specify` 结构嵌入到新项目中。
- **FR-002**: 系统必须提供一个能够解析 `plan.md` 文件的 "Planner Agent"。
- **FR-003**: Planner Agent 必须能够根据功能元数据自动创建 Git 分支。
- **FR-004**: Planner Agent 必须能够根据技术计划中定义的目录结构和占位文件进行生成。
- **FR-005**: Planner Agent 必须能够根据计划中列出的任务自动生成/更新 `tasks.md`。
- **FR-006**: SpecKit 命令必须通过 ClawSharp CLI 公开。
- **FR-007**: **交互式确认机制**：当 Planner Agent 检测到 `plan.md` 发生显著变化或保存时，必须主动询问用户是否执行脚手架生成，而不是静默执行或强制手动命令。

### 关键实体 *(如果功能涉及数据)*

- **SpecKit**: 核心配置和模板包。
- **ProjectScaffolder**: 负责生成项目结构的内核服务。
- **PlannerAgent**: 负责计划到执行转化的专门 AI 助手。
- **FeatureContext**: 与功能关联的元数据（名称、分支、状态、规范路径、计划路径）。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 开发者可以通过单个 CLI 命令初始化一个“规范就绪”的项目。
- **SC-002**: 从“计划敲定”到“准备编码”（分支已创建，文件已就绪）的时间减少到 AI 处理的 30 秒以内。
- **SC-003**: 技术计划中列出的 100% 文件都能由自动脚手架正确创建。
- **SC-004**: 在初始脚手架生成后，`tasks.md` 始终与 `plan.md` 的里程碑保持 100% 同步。

## 假设

- **Git 集成**: 我们假设本地环境已安装并配置了 `git`。
- **模板来源**: 我们假设 `.specify` 模板是 ClawSharp 发行版的一部分，或者可以从中央仓库获取。
- **文件系统权限**: 应用程序对项目工作空间具有读写权限。
- **LLM 能力**: 我们假设 LLM 能够可靠地解析基于 Markdown 的计划并提取结构化数据（JSON/YAML）。
- **分支命名**: 除非另有配置，否则默认分支命名遵循 `[id]-[short-name]` 模式。
