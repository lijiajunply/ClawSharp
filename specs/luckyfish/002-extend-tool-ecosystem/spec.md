# 功能规范：扩展 AI 工具生态系统

**功能分支**：`luckyfish/002-extend-tool-ecosystem`  
**创建日期**：2026-03-30  
**状态**：草案  
**输入**：用户描述：“我现在想加入更多的 Tool ,使其可以让 AI 完成更多的工作,查看更多类型的数据”

## 用户场景与测试 *(必填)*

### 用户场景 1 - 网页研究与内容检索 (优先级: P1)

作为研究人员，我希望 AI 能够从特定的网页 URL 或搜索结果中获取内容，以便它能提供最新信息并引用外部来源。

**优先级说明**：获取外部信息是扩展 AI 处理能力（超出其训练数据或本地文件）的基础能力。

**独立测试**：可以通过要求 AI “总结 https://example.com 的内容”来测试，并确认它使用了 `web_browser` 工具。

**验收场景**：

1. **给定** 一个有效的 URL，**当** AI 调用浏览器工具时，**那么** 系统返回页面的文本内容（支持由 JavaScript 渲染的内容）。
2. **给定** 一个受限的 URL（通过权限集），**当** AI 尝试获取它时，**那么** 系统拒绝该请求。

---

### 用户场景 2 - 结构化数据分析 (优先级: P2)

作为数据分析师，我希望 AI 能够直接读取和解释 CSV 或 Excel 文件，以便它可以进行计算、生成摘要或转换数据。

**优先级说明**：用户经常处理电子表格；允许 AI 原生“看到”这种数据类型可以提高其处理业务任务的效用。

**独立测试**：可以通过提供一个 CSV 文件并要求 AI “计算 X 列的平均值”来全面测试。

**验收场景**：

1. **给定** 一个本地 CSV 文件，**当** AI 调用 `csv_read` 工具时，**那么** 系统返回数据的结构化表示（如 JSON 对象数组）。
2. **给定** 一个大型 CSV 文件，**当** AI 读取它时，**那么** 系统提供分页或摘要视图，以避免上下文窗口溢出。

---

### 用户场景 3 - 版本控制集成 (优先级: P3)

作为开发人员，我希望 AI 能够与 Git 交互，以便它可以检查仓库状态、查看历史记录或准备提交，而无需我手动运行 shell 命令。

**优先级说明**：虽然 `shell_run` 可以执行 git 命令，但专用工具为 AI 提供了更好的结构化输出、安全性和可发现性。

**独立测试**：可以通过询问 “此仓库最近有哪些更改？”并确认它使用了 `git_log` 工具来测试。

**验收场景**：

1. **给定** 一个 Git 仓库，**当** AI 调用 `git_status` 工具时，**那么** 它会收到修改、添加和删除文件的列表。
2. **给定** 一个特定的提交，**当** AI 请求 diff 时，**那么** 该工具返回简洁的 diff 输出。

---

### 用户场景 4 - 文档文本提取 (优先级: P3)

作为学生，我希望 AI 能够读取 PDF 文档，以便它可以帮助我学习或总结论文。

**优先级说明**：将 AI 可以看到的数据类型扩展到包括常见的文档格式。

**独立测试**：可以通过要求 AI 总结本地 PDF 文件来测试。

**验收场景**：

1. **给定** 一个 PDF 文件，**当** AI 调用 `pdf_read` 工具时，**那么** 系统提取并返回文本内容。

---

### 边缘情况

- **大文件处理**：系统如何处理 100MB 的 CSV 或 50 页的 PDF？（假设：分页读取或自动摘要）。
- **网络超时**：系统如何处理响应缓慢或无响应的 Web 服务器？（要求：可配置的超时时间）。
- **二进制/非文本数据**：当使用文本工具处理非文本文件时，系统如何处理？（要求：优雅的错误处理）。

## 功能要求 *(必填)*

### 核心要求

- **FR-001**：系统必须提供 `web_browser` 工具，支持使用无头浏览器（如 Playwright）获取包含 JavaScript 渲染内容的网页。
- **FR-002**：系统必须提供 `csv_read` 工具，将 CSV 文件解析为结构化的 JSON 数据。
- **FR-003**：系统必须提供一组内置的 `git_ops` 工具（至少包括：status, log, diff）用于仓库交互。
- **FR-004**：系统必须提供 `pdf_read` 工具，从 PDF 文档中提取纯文本。
- **FR-005**：所有新工具必须严格遵守 `ToolPermissionSet` 限制（读取根目录、写入根目录和能力位）。
- **FR-006**：结构化数据工具（CSV/JSON）必须支持分页或限制返回量，以防止上下文溢出。
- **FR-007**：系统必须允许在 `ToolRegistry` 中注册这些工具。

### 关键实体

- **ToolInvocationResult**：(现有) 返回给 AI 的有效载荷，现在必须支持更丰富的结构化数据（例如 CSV 的 JSON 对象）。
- **ToolCapability**：(现有) 可能需要新增 `NetworkAccess`（用于网页）和 `VersionControl`（用于 Git）等标志，或者复用现有能力位。

## 验收标准 *(必填)*

### 可衡量的结果

- **SC-001**：AI 能够在单次对话中成功获取并总结远程网页（包括动态加载的内容）。
- **SC-002**：AI 能够正确识别并处理具有至少 1000 行数据的 CSV 文件（通过分页）。
- **SC-003**：新工具的执行开销（不包括外部网络/IO 延迟）应在 500 毫秒以内。
- **SC-004**：在缺少所需权限时，100% 的新工具都能正确触发拒绝逻辑。

## 假设与依赖

- **本地执行**：PDF 和 CSV 处理将本地完成，不依赖外部 API。
- **浏览器环境**：系统假设运行环境支持启动无头浏览器（可能需要安装相应的浏览器驱动）。
- **工具内置**：为了性能和安全性，所有新工具都将作为 C# 内置类在 ClawSharp 内核中实现。
- **依赖库**：项目可以引入标准的 .NET 库（如 `Playwright`, `CsvHelper`, `PdfPig`）。
